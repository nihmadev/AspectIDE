# Project Architecture Rules

Делай изменения качественно, оптимизированно, красиво и удобно. Не халтурь, не лей воду, не ломай чужие незавершенные изменения.

Эти правила применяются пропорционально задаче и текущей архитектуре Lux IDE. Не запускай большой рефакторинг ради мелкого фикса, но не добавляй новый долг, если можно аккуратно вынести ответственность сразу.

## Architecture Model

- Держи layered + domain-first направление: domain/pure logic, application/use-cases, infrastructure/IO/Tauri/network/files/AI providers, UI/React presentation.
- Domain-логика не должна зависеть от IO, Tauri, React, сети или файловой системы.
- Application оркестрирует use-cases и зависит от domain-моделей, а infrastructure адаптирует их к реальному IO.
- UI должен работать через typed API/helpers, а не через случайные внутренние детали соседних модулей.
- Не вводи циклические зависимости и скрытые runtime-связи.

## Decomposition

- Не добавляй код в модуль, если он не соответствует его ответственности.
- Один модуль должен иметь одну основную причину для изменения.
- Не раздувай `apps/desktop/src-tauri/src/lib.rs`: новые Tauri-команды и сервисы группируй по доменам и выноси в модули по мере касания.
- Новые модули должны иметь явный typed public API. Не протаскивай внутренние детали через случайные импорты.
- Extraction должен снижать связанность, устранять дублирование или изолировать IO. Не дроби ради косметики.
- Нет искусственного лимита строк. Большой файл плох не числом строк, а смешением ответственностей, сложностью тестирования и расширения.

## Domain Discipline

- Бизнес-логика должна быть pure там, где это возможно.
- Ошибки в публичных API домена должны быть типизированы; не прячь смысл ошибки в строках без необходимости.
- Инварианты фиксируй типами, newtype-ами и явными preconditions/postconditions, когда это реально защищает поведение.
- Невозможные состояния по возможности делай невозможными на уровне типов.

## Frontend

- Контейнеры управляют state/effects, leaf-компоненты рендерят UI, domain helpers живут вне React.
- Минимизируй подписки на store: подписывайся на нужные selectors и выноси тяжелый render в memoized leaf-компоненты, когда это снижает стоимость rerender.
- Тяжелые поверхности вроде AI chat, history, tools, settings, editor/workspace panels держи в feature-папках или отдельных компонентах.
- Не смешивай unrelated экраны в одном CSS-контексте. Глобальные CSS-секции группируй по feature.
- Избегай скрытых side effects в hooks.

## AI Runtime

- AI runtime развивай как набор независимых подсистем: providers, streaming, context budget, tool registry, tool executors, approvals, checkpoints, diagnostics, history.
- Каждый tool должен иметь typed input/output, понятный risk level, timeout, output limits и нормальную ошибку.
- Опасные операции требуют preview или явного безопасного режима.
- Context assembly должен быть объяснимым: что добавлено, что отброшено, почему и в каком приоритете.
- Никаких silent fallback моделей или скрытой магии в выборе провайдера.

## Rust / Tauri

- Crate-level правила объявляй только в `main.rs`, `lib.rs` и entry-points:

```rust
#![deny(clippy::pedantic)]
#![deny(clippy::nursery)]
#![allow(clippy::missing_errors_doc)]
```

- Локальные `allow` допустимы только точечно и с понятной причиной, особенно вокруг framework ABI/macros, где переписывание рискованнее предупреждения.
- Предпочитай идиоматичный Rust: `.is_empty()` вместо `len() == 0`, `&[T]` вместо `&Vec<T>`, ранние выходы вместо глубокой вложенности.
- Минимизируй clone и alloc. Не используй `unwrap` в production-коде, не игнорируй `Result`, избегай скрытых panic.
- Shared state не должен становиться глобальной помойкой. IO helpers, commands, diagnostics и AI/file/web tools держи разделенными.

## Performance

- Осознавай сложность алгоритмов и memory pressure, особенно в AI runtime, history, streaming, search и больших UI-list surfaces.
- Избегай O(n^2), лишних промежуточных коллекций и лишних clone без причины.
- Сначала сохраняй корректность и понятные границы, потом оптимизируй горячие участки.

## Verification

- После изменений запускай минимально достаточные проверки: build/typecheck/tests/clippy по затронутой зоне.
- После Rust-правок запускай релевантный `cargo clippy` с pedantic/nursery gate; если полный workspace пока не чистый, укажи конкретный оставшийся долг.
- Если цель закрыта не полностью, называй оставшийся архитектурный долг конкретно.

## Absolute Rules

- Никакой скрытой магии.
- Никаких временных костылей без явной пометки.
- Никаких silent fallback.
- Никакого смешивания слоев без причины.
- Никакого подхода “работает и ладно”.
