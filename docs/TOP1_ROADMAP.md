# Lux IDE → Top-1 Roadmap

_Generated from an 8-zone full-project audit (agent-loop, context engine, editor, frontend UX, Rust foundation, performance, release/reliability, competitive gaps). Impact = how much it moves toward #1; Effort S=hours / M=days / L=week+ / XL=multi-week._

## Verdict

_Note: the strategic pillars below are cross-referenced against the grounded per-zone findings in the Appendix; where they differ, trust the Appendix._

Honest standing: Lux's agent surface is genuinely competitive with Cursor/Windsurf — subagents, plan cards, checkpoints, MCP, skills, persistent memory, code graph, web research, and a Monaco diff-review flow, on a Rust core with 522 passing tests that has just been security-hardened (54 confirmed findings fixed through v1.0.9/1.0.10). But Lux is not yet a daily-driver IDE: there is no inline tab-completion at all (the highest-frequency AI feature in every top tool), no project-wide replace, only 8 frontend test files against a very large TS surface, and nearly every npm dependency floats at "latest", making shipped builds non-reproducible. The single most important thing to fix is shipping inline ghost-text completions — without them users keep their primary editor elsewhere and Lux stays "the agent app beside my IDE" rather than the IDE. Cheapest critical wins right behind it: pin dependencies and add a permission-enforcement regression suite so the C1/C2 class of native-loop bypass can never silently return.

## Appendix — per-zone audit

### Zone: `AI agent loop — native Rust turn loop (ai_turn.rs dispatch/loop, ai_prompt.rs, ai_chat_backend.rs + ai_anthropic.rs transport, ai_compaction.rs, ai_session.rs, ai_permissions.rs, ai_checkpoint.rs)`

**Strengths:** Cancellation correctness is exceptional: FIFO-evicting cancel registry, race_cancel polls Stop during connect, silent stream stalls, and retry backoff (ai_chat_backend.rs:1652), and the loop checks cancellation at every boundary — before rounds, post-stream, between tools, inside subagents.; Retry engine rivals Claude Code: per-status budgets (429/5xx get 10 attempts, edge-403 gets 4), Retry-After honored, linear backoff ladder, live TurnRetry UI events, and mid-stream error frames (Anthropic typed `error` events and OpenAI-compatible {"error":...} on a 200 stream) correctly fail the turn instead of finalizing truncated output.; Security layering at the Rust boundary: tool allowlist built from the exact definitions sent (F4, forged calls rejected), deny>ask>allow permission engine with compound-shell segment scanning, basename path matching, Bash/Shell aliasing, rules loaded from trusted SettingsStore not the renderer (ai_permissions.rs).; Read-before-edit guard with mtime+size staleness fingerprints (ai_session.rs ReadStamp) plus same-batch read+edit rejection (F6) — stronger than most shipping agents, which only track 'was read ever'.; Turn resource budgets: 32k-char per-tool clamp, 600KB/200-call per-turn ceilings with model-visible truncation notices; recovery synthesis (tool_choice:"none") guarantees a real final answer instead of a bare 'Done.'; Mid-turn steering is real: ai_inject_message stages user messages in turn-scoped queues drained at round boundaries AND before a natural finish, with UI bubble events — genuine Claude Code steering parity.; Deterministic 5-phase plan quality gate (assess_plan_quality: decompose/alternatives/critique/verification/rollback with coaching nudges) — structurally beyond what Claude Code's plan mode does.; Transport hygiene: UTF-8 chunk-boundary reassembly, inline <think> extraction with cross-chunk tag splitting, non-SSE gateway fallback, SSE buffer cap, clean Anthropic↔OpenAI translation seam (ai_anthropic.rs) keeping the whole loop protocol-agnostic.; Compaction summary contract: 9 required markdown sections validated as present AND non-empty before storage, tail-preserving transcript so latest direction/errors always reach the summarizer (ai_compaction.rs).; Prompt assembly fail-safes: unknown agent mode defaults to read-only with an explicit warning, workspace-root redaction, per-section byte budgets, shell disclosure tied to the same cfg!(windows) as the real executor (ai_prompt.rs).

**Quick wins:** Add a message-level cache_control breakpoint on the last history message each round for anthropic_cache providers — the single highest cost/latency win per line of code (verify via the already-parsed cache_read_input_tokens).; Surface malformed tool-call JSON as an explicit recoverable error (with the parse failure position) instead of silently coercing to {} in execute_tool (ai_turn.rs:1676).; Raise/parameterize Anthropic DEFAULT_MAX_TOKENS (currently 8192 in ai_anthropic.rs) per model so large multi-file edits stop truncating at 'length'.; Emit subagent ToolCallStarted/Completed events tagged with the agent id so a running Task shows live progress instead of a frozen 'running-tools' spinner.; Persist session goal/todos into the existing ai-chat-history.json document so the task rail survives an app restart.; Compare cumulative usage_prompt against a model context limit passed in TurnInput and emit a UI warning event when a turn crosses ~80% — groundwork for full in-turn compaction.

- 🔴 CRIT [M] **Extended thinking is never enabled on the native Anthropic protocol** — ai_anthropic.rs to_anthropic_request explicitly drops reasoning_effort/reasoning ('Anthropic would reject them') and never maps effort → thinking:{type:enabled,budget_tokens}; DEFAULT_MAX_TOKENS hard-caps output at 8192 (Claude 4.x supports far more); convert_assistant replays history without thinking blocks, which Anthropic requires preserved across tool-use rounds. The receive side is already built (AnthropicStreamAccumulator handles thinking_delta) — the send side just never turns it on. Claude models run with thinking OFF while Claude Code runs them interleaved: a direct reasoning-quality deficit on the flagship models.  
  → Map the selected effort to thinking.budget_tokens in to_anthropic_request; raise/parameterize max_tokens per model; capture thinking blocks (with signatures) in the turn's message log and replay them in convert_assistant; evaluate the interleaved-thinking beta header.
- 🔴 CRIT [S] **Prompt caching covers only the system prompt — every round re-bills the whole conversation** — build_system_message (ai_turn.rs:154) sets one cache_control breakpoint on the ~15KB system prompt; the per-round messages array — history plus accumulating tool results, which dominates tokens from round 3 onward — has no breakpoints. On Anthropic, every tool round of a 30-round turn pays full input price and full TTFT for the entire conversation. run_subagent has the same shape. Claude Code moves a breakpoint to the newest message each round so round N+1 reads round N from cache; usage_cached already parses cache_read_input_tokens, so the win would be immediately measurable.  
  → Add ephemeral cache_control breakpoints on the last message (and optionally the tools array) when assembling each round's payload for anthropic_cache providers — a small change in ai_turn.rs payload assembly or to_anthropic_request with outsized cost/latency payoff.
- 🔴 CRIT [M] **No in-turn context-window management — long turns grow until they die** — Compaction runs only pre-turn from TS (AiChatPanel.tsx awaits compactChatHistory before send; ai_compaction.rs is just the summarizer). Inside ai_run_turn the messages vec grows across up to 128 rounds with only char-based guards (32k chars/tool, 600KB/turn ≈ 150k+ tokens) that know nothing about the actual model context limit, and a provider 'context length exceeded' mid-turn surfaces as a terminal TurnError with no compact-and-retry. Claude Code auto-compacts mid-session; here a long agentic task on a smaller-window model hard-fails.  
  → Thread the model's context-token limit into TurnInput, compare against per-round usage_prompt, and when nearing the limit prune oldest tool outputs / inject a native checkpoint summary (reuse ai_compaction_summary) inside the loop; catch context-overflow errors and retry once after in-place compaction.
- 🟠 HIGH [M] **Strictly serial tool execution; Task subagents block the parent and are invisible** — execute_tool is awaited per call in order (ai_turn.rs:1168–1336): a batch of 5 independent Reads costs 5× latency, and parallel_tool_calls is never set on the payload. Task blocks the whole parent turn for up to 16 subagent rounds while interactive=false suppresses every UI event — the user stares at a frozen 'running-tools' phase — and only one subagent can ever run at a time. Claude Code executes read-only batches concurrently and runs multiple background subagents with streamed progress.  
  → Classify calls read-only vs mutating (shell classification already exists in ai_shell_safety) and join independent read-only calls concurrently; emit subagent ToolCallStarted/Completed events tagged with agent_id; allow Task to run in background with a completion event and support N concurrent subagents.
- 🟠 HIGH [M] **No hooks / lifecycle extension points anywhere in the loop** — Nothing fires user-defined logic on pre/post tool use, prompt submit, or turn end — the only extension surfaces are permission rules (ai_permissions.rs) and MCP servers. Claude Code's PreToolUse/PostToolUse/Stop hooks are how teams enforce formatters, linters, commit policies, and custom guards deterministically; Lux has no equivalent seam in execute_tool's dispatch.  
  → Add a workspace hooks config (e.g. .lux/hooks.json: matcher + command + timeout) evaluated in execute_tool before/after dispatch with allow/deny/mutate-output semantics, reusing the existing approval/permission plumbing and event channel.
- 🟠 HIGH [M] **Checkpoint/rewind is manual, in-memory, and capped — protection depends on model discipline** — ai_checkpoint.rs stores snapshots in a process-global in-memory HashMap (max 24 checkpoints, 80 files, 1MB/file), cleared on workspace close and lost on restart/crash — exactly when rollback matters most. Nothing auto-checkpoints before a turn's first mutating tool; prompts/core.txt merely instructs the model to 'Checkpoint before risky work'. There is no conversation-linked rewind (restore files + history to turn N) like Claude Code's automatic per-edit rewind.  
  → Auto-snapshot touched files on a turn's first Write/StrReplace/PatchEngine/Delete (those arms already resolve the paths), persist snapshots to app-data disk keyed by session+turn, and surface per-turn rewind in the chat UI.
- 🟠 HIGH [M] **Subagent system is shallow: 3 hardcoded types, no custom agents, no resume, approvals auto-reject** — run_subagent recognizes only generalPurpose/codeReviewer/explorer via a match on read_only (ai_turn.rs:4345); there is no user-defined agent registry (Claude Code loads .claude/agents definitions with per-agent prompt/tools/model). Task resume is explicitly rejected (ai_turn.rs:3903), any approval-requiring tool auto-rejects for subagents (require_tool_approval with interactive=false), rounds cap at 16, and subagent token usage is never accounted into TurnUsage.  
  → Load agent definitions from the workspace (name, system prompt, tool allowlist, model, max rounds); forward subagent approval requests to the parent UI instead of auto-rejecting; report per-subagent usage into TurnUsage.
- 🟡 MED [M] **A stream failure after the first token kills the entire turn** — completion_streaming retries only the connection phase (ai_chat_backend.rs:407–419, by design); a socket drop or provider stream error at round N returns Err and ai_run_turn emits TurnError and exits (ai_turn.rs:1028–1043), discarding the in-loop messages vec even after 20 successful tool rounds. The round's request is replayable (nothing was committed by the model call itself), but there is no round-level retry — the user must manually re-send and the turn context is rebuilt from frontend history.  
  → On mid-stream failure with prior successful rounds, retry the round's model call under a bounded budget (emitting a UI reset-delta event for the partial text) before surfacing TurnError.
- 🟡 MED [M] **Session/turn state is volatile — no resume after restart or crash** — Goals, todos, and read-file fingerprints live in process-global in-memory maps (ai_session.rs); a restart mid-task loses the pinned goal and task list (chat history survives via ai-chat-history.json, but the goal/todo rail resets and an interrupted turn cannot be resumed). Claude Code persists transcripts and supports resuming a session with intact working state.  
  → Persist session goal/todos alongside the chat-history document and rehydrate on load; add a 'continue interrupted turn' affordance that replays the stored history plus a synthetic 'the previous turn was interrupted after X' user message.
- 🟡 MED [S] **Malformed tool-call JSON silently coerces to {} + 5.5k-line monolithic dispatch** — execute_tool parses arguments with serde_json::from_str(...).unwrap_or_else(|_| json!({})) (ai_turn.rs:1676), so a trailing comma or truncated JSON from a weaker model becomes a misleading 'path is required' error instead of 'your arguments were invalid JSON at …', costing an extra repair round and mistraining the model's fix. Separately, ai_turn.rs at 5,494 lines holds the loop, ~50 tool arms, subagents, approvals, and the plan gate in one file, which slows every extension listed above.  
  → Return an explicit recoverable error naming the JSON parse failure and offending fragment; then split execute_tool into per-domain modules (fs, shell, browser, graph, orchestration) so new capabilities (hooks, parallel dispatch) land safely.
