# Installer and Distribution Policy

AspectIDE ships as a native desktop product, not as a loose archive.

## Targets

- Windows 10/11: x64 and ARM64 via NSIS/WiX-class native installers.
- macOS: Intel and Apple Silicon via signed `.dmg`; `.pkg` can be added for managed environments.
- Linux: AppImage, `.deb`, and `.rpm`.

## Windows Install

The Tauri bundle config enables an NSIS installer with custom AspectIDE hooks:

- branded installer icon and localized language selector;
- per-user or machine install mode;
- Start Menu/Desktop shortcuts through the native installer;
- protocol registration for `aspect://`;
- registry cleanup on uninstall.

The uninstall hook asks whether to remove AspectIDE caches and data. This full cleanup removes app-owned config, caches, logs, indexes, recent workspaces, and downloaded extensions. It must never delete user project directories.

## macOS Install

Production releases must use Apple Developer ID signing and notarization before public distribution. App support, caches, logs, and extension data live under platform-standard app directories so a cleanup helper can remove AspectIDE-owned data without touching projects.

## Linux Install

Linux packages must register desktop entries, icons, and MIME/protocol associations through package scripts. Full cleanup must remove only AspectIDE-owned config/cache/data directories.

## Updates

The updater plugin is linked in the desktop shell, but the default open-source configuration does not expose update permissions or generate updater artifacts until a signed release channel exists. Production update manifests must use a real HTTPS endpoint, a non-empty public key generated by the Tauri signer, and separate channels: `stable`, `beta`, and `nightly`. Update migrations must be rollback-safe and preserve user projects.

Before enabling updater artifacts for a release, verify all of the following:

- `bundle.createUpdaterArtifacts` is enabled only in release configuration.
- `plugins.updater.pubkey` contains the public key matching the signing private key.
- `plugins.updater.endpoints` points at the production update manifest host, not a local or placeholder domain.
- `updater:default` or narrower updater permissions are added only when the frontend exposes a real update workflow.

## GitHub Releases

Release installers are built by `.github/workflows/release.yml` from a clean git tag. The tag must match the app version in both `package.json` and `apps/desktop/src-tauri/tauri.conf.json`.

Release flow:

1. Update both version fields to the same semantic version.
2. Push a tag such as `v1.0.0`, or run the `Release Installers` workflow manually with the same tag.
3. The workflow verifies that source config does not commit signing/updater secrets, prepares a signed release config in the CI checkout, runs quality gates, builds native Tauri bundles on Windows, macOS, and Linux runners, and publishes installer assets to the GitHub Release.

Required GitHub release secrets:

- `WINDOWS_CERTIFICATE_PFX_BASE64`, `WINDOWS_CERTIFICATE_PASSWORD`, `WINDOWS_CERTIFICATE_THUMBPRINT`
- `APPLE_CERTIFICATE_P12_BASE64`, `APPLE_CERTIFICATE_PASSWORD`, `APPLE_KEYCHAIN_PASSWORD`, `APPLE_ID`, `APPLE_PASSWORD`, `APPLE_PROVIDER_SHORT_NAME`, `APPLE_SIGNING_IDENTITY`
- `TAURI_SIGNING_PRIVATE_KEY`, `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`, `TAURI_UPDATER_PUBLIC_KEY`, `TAURI_UPDATER_ENDPOINTS`

`TAURI_UPDATER_ENDPOINTS` must list production HTTPS manifest URLs for all public channels: `stable`, `beta`, and `nightly`. The release verifier rejects local, placeholder, duplicated, or unsigned-updater configuration before installers are built.

Published assets include Windows NSIS installer, macOS app/dmg bundles, Linux AppImage/deb/rpm packages, `SHA256SUMS.txt`, and `release-manifest.json`. Public macOS releases still require Apple Developer ID signing and notarization secrets before they should be treated as fully trusted distribution builds.
