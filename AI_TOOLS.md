# AI Tools

Короткий реестр для AI Chat. Цель: разделить callable tools, которые модель вызывает явно, и функционал платформы, который ускоряет/защищает работу tools автоматически.

## Правила для всех tools

| Правило | Требование |
|---|---|
| Один tool - одно действие | Не смешивать поиск, редактирование и запуск команд в одном инструменте. |
| Явный input/output | Каждый tool должен иметь понятные параметры и предсказуемый результат. |
| Безопасность по умолчанию | Удаление, перезапись, shell-команды, GitHub-мутации и действия в UI требуют защитных проверок. |
| Bulk-вызовы | Runtime должен поддерживать запуск tools пачкой: параллельно для независимых действий и последовательно для зависимых шагов. |
| Машиночитаемые ошибки | Ошибка должна возвращать причину, код/тип ошибки и возможный следующий шаг. |
| Лог действий | Важные изменения должны быть видимы пользователю: что изменено, где и почему. |

## Встроенные инструменты Cursor

| Tool | Назначение | Когда использовать |
|---|---|---|
| `SemanticSearch` | Семантический поиск по коду. | Когда вопрос звучит как `как работает X?`, `где реализовано Y?`, `что отвечает за Z?`. |
| `Grep` | Быстрый текстовый/regex-поиск через ripgrep. | Когда известен точный текст, имя функции, строка ошибки, ключ или паттерн. |
| `Glob` | Поиск файлов по маске. | Когда нужны файлы определенного типа, имени или расположения. |
| `Read` | Чтение файлов, включая изображения. | Перед любым редактированием, ревью или анализом конкретного файла. |
| `Write` | Создание или полная перезапись файла. | Для новых файлов или полной генерации артефакта. Не использовать для точечных правок. |
| `StrReplace` | Точечная замена в файле. | Когда нужно изменить небольшой фрагмент без переписывания всего файла. |
| `Delete` | Удаление файла. | Только когда файл точно лишний и пользователь/контекст это подтверждает. |
| `Shell` | Команды терминала: git, npm, build, test, scripts. | Для сборки, тестов, диагностики, CLI-операций и локальных утилит. |
| `ReadLints` | Диагностика линтера по файлам. | После правок кода или при разборе ошибок качества. |
| `EditNotebook` | Редактирование ячеек Jupyter. | Для `.ipynb`: код, markdown, порядок и содержимое ячеек. |
| `TodoWrite` | Список задач текущей сессии. | Для многошаговых задач, где важно показать прогресс и не потерять этапы. |
| `AskQuestion` | Структурированный вопрос с вариантами ответа. | Когда без выбора пользователя нельзя безопасно продолжать. |
| `Task` | Запуск под-агентов: explore, shell, debugger, frontend-impl и др. | Для параллельного исследования, изолированной диагностики или специализированной работы. |
| `Await` | Ожидание фонового shell-процесса. | Когда сборка, тесты, сервер или долгий скрипт уже запущены. |
| `WebSearch` | Поиск в интернете. | Для актуальной информации, документации, релизов, ошибок и внешних источников. |
| `WebFetch` | Загрузка и разбор URL. | Когда известна конкретная ссылка и нужно извлечь содержимое страницы. |
| `GenerateImage` | Генерация изображений по описанию. | Для растровых ассетов, мокапов, иллюстраций, текстур и визуальных вариантов. |
| `SwitchMode` | Переключение режима `agent` / `plan`. | Когда нужно перейти от планирования к исполнению или наоборот. |
| `CallMcpTool` | Вызов tool из MCP-сервера. | Для интеграций: БД, браузер, документы, внешние сервисы, кастомные серверы. |
| `FetchMcpResource` | Чтение ресурсов MCP. | Когда MCP-сервер отдает контекст: схемы, файлы, справочники, состояние сервиса. |

## Дополнительные callable tools

Это не замена базовым tools, а слой ускорения и качества. В список входят только инструменты, которые дают новый тип контекста, анализа или проверки.

| Tool | Назначение | Почему не дубль |
|---|---|---|
| `FastContext` | Собрать короткий пакет контекста под задачу: релевантные файлы, символы, diff, ошибки, тесты, правила проекта и риски. | Оркестратор: сам выбирает, какие базовые tools вызвать, и возвращает готовый контекст вместо сырого набора файлов. |
| `RepoMap` | Сжатая карта проекта: стек, entrypoints, важные директории, конфиги, scripts, env-схемы. | `Glob` только находит файлы, а `RepoMap` объясняет структуру и точки входа. |
| `SymbolContext` | Единый инструмент по символам: определение, использования, сигнатура, импорты, call sites. | Заменяет набор мелких tools вроде `GoToDefinition`, `FindReferences`, `SymbolSearch`, `CallGraph`. |
| `RelatedFiles` | Найти связанные файлы: тесты, стили, типы, stories, routes, API, schemas, mocks. | Не ищет текст, а понимает связи внутри проекта. |
| `GitContext` | Структурированный git-контекст: branch, status, staged/unstaged diff, base branch, recent commits. | `Shell` может это вывести, но `GitContext` возвращает короткую модель состояния без ручного парсинга. |
| `DiagnosticsContext` | Общая диагностика: lints, TypeScript, build, test errors, browser console/network errors. | Шире, чем `ReadLints`: собирает все текущие ошибки в один список причин. |
| `TestContext` | Найти релевантные тесты, последние падения, покрываемые сценарии и минимальную команду запуска. | `Shell` запускает команды, а `TestContext` выбирает правильные тесты и объясняет, зачем они нужны. |
| `FailureAnalyzer` | Разобрать лог команды, теста, build, CI или browser console и выделить root cause. | Не просто показывает лог, а сжимает шум до причины, файла, строки и следующего действия. |
| `DocsContext` | Достать документацию по установленным версиям библиотек, framework APIs и локальным README. | `WebSearch` ищет в интернете, а `DocsContext` привязывает ответ к версиям и зависимостям проекта. |
| `MemoryContext` | Найти проектные решения, договоренности, пользовательские предпочтения и прошлые похожие задачи. | Не файловый поиск: работает с долговременной памятью проекта и команды. |
| `ImpactAnalysis` | Оценить blast radius перед правкой: затронутые модули, контракты, тесты, миграции, UI-сценарии. | Используется до изменений, чтобы не чинить один файл и не ломать соседние. |
| `ReviewDiff` | Проверить текущий diff на баги, регрессии, пропущенные тесты, опасные изменения и лишний шум. | Это финальный quality gate, а не еще один способ редактировать файл. |

## Функционал платформы

Это не tools. Модель не должна выбирать их как отдельное действие над проектом. Это возможности IDE/runtime, которые делают callable tools быстрее, безопаснее и умнее.

| Функция | Что дает | Кому помогает |
|---|---|---|
| `WorkspaceIndex` | Инкрементальный индекс файлов, символов, imports/exports, configs и зависимостей. | `FastContext`, `RepoMap`, `SymbolContext`, `RelatedFiles`. |
| `ActiveContext` | Текущие вкладки, выделение, cursor position, активный терминал, последние правки, видимые ошибки. | `FastContext`, `DiagnosticsContext`, `FailureAnalyzer`. |
| `RulesContext` | Автоподхват `.cursorrules`, `AGENTS.md`, `CLAUDE.md`, README и IDE-rules. | Все tools, которые принимают решения по коду. |
| `Checkpoint` | Снапшот перед AI-правками, просмотр diff и откат только изменений агента. | `Write`, `StrReplace`, `Delete`, multi-file edits. |
| `PermissionPolicy` | Централизованные правила разрешений: auto-allow, ask, deny для опасных действий. | `Shell`, `Delete`, GitHub write-tools, Computer/Browser actions. |
| `SecretGuard` | Проверка секретов в diff, logs, screenshots, web/GitHub payloads и shell output. | `Shell`, `GitHub*`, `WebFetch`, `Browser*`, `ComputerScreenshot`. |
| `ProcessManager` | Учет долгих процессов: dev servers, watch-tests, background jobs, logs, restart/kill. | `Shell`, `Await`, `DiagnosticsContext`, `FailureAnalyzer`. |
| `PatchEngine` | Preview, partial apply, conflict handling и атомарное применение multi-file patches. | `Write`, `StrReplace`, `ReviewDiff`, `Checkpoint`. |
| `ContextBudgeter` | Выбор, сжатие и приоритизация контекста под лимит модели. | `FastContext`, `MemoryContext`, `DocsContext`. |

## Bulk / Batch-вызовы

Batch-вызовы - это функция runtime, а не отдельный tool. Она нужна, чтобы AI не тратил время на ручной поочередный запуск независимых tools и мог явно описывать цепочки зависимостей.

| Режим | Как работает | Пример |
|---|---|---|
| `parallel` | Все независимые tools запускаются разом. | Одновременно прочитать 5 файлов, получить git status, diagnostics и package context. |
| `sequence` | Tools запускаются строго по очереди, каждый следующий может использовать результат предыдущего. | Найти файл через `SemanticSearch`, прочитать его через `Read`, затем получить `RelatedFiles`. |
| `dag` | Смешанный режим: независимые ветки идут параллельно, зависимые ждут нужные результаты. | `FastContext`: параллельно `RepoMap`, `GitContext`, `DiagnosticsContext`; потом `RelatedFiles` по найденным файлам. |

Минимальные параметры batch-вызова:

| Параметр | Назначение |
|---|---|
| `mode` | `parallel`, `sequence` или `dag`. |
| `calls` | Список tool-вызовов с именем tool, input и локальным id. |
| `dependsOn` | Для `dag`: какие вызовы должны завершиться перед текущим. |
| `concurrency` | Ограничение одновременных вызовов, чтобы не перегружать IDE/терминал/API. |
| `stopOnError` | Остановить весь batch при первой ошибке или продолжать независимые ветки. |
| `timeoutMs` | Общий или индивидуальный timeout. |
| `resultMode` | Вернуть все результаты, только summary или только ошибки. |

Правило выбора: если tools не зависят друг от друга, использовать `parallel`; если результат одного нужен следующему, использовать `sequence`; если есть обе ситуации, использовать `dag`.

### Рекомендуемый формат `FastContext`

```md
## FastContext
- Intent: что пользователь хочет сделать
- Relevant files: минимум нужных файлов и почему они выбраны
- Symbols: функции, классы, компоненты, типы, API
- Git state: важный diff/status/branch
- Diagnostics: ошибки, которые уже есть
- Related tests: что запускать для проверки
- Project rules: локальные соглашения и ограничения
- Risks: что можно сломать
- Next tools: какие tools вызвать дальше
```

### Минимальный v1

Если внедрять постепенно, начать с этих tools:

| Приоритет | Tool | Причина |
|---|---|---|
| P0 | `FastContext` | Самый большой прирост скорости и качества ответов. |
| P0 | `GitContext` | AI сразу понимает текущие изменения и не трогает чужое. |
| P0 | `DiagnosticsContext` | Ошибки становятся частью контекста до правок. |
| P1 | `SymbolContext` | Безопасные изменения в коде без слепого grep. |
| P1 | `RelatedFiles` | Быстро находит тесты и соседние файлы. |
| P1 | `FailureAnalyzer` | Ускоряет отладку логов и CI. |
| P2 | `DocsContext` | Снижает галлюцинации по API библиотек. |
| P2 | `ReviewDiff` | Дает контроль качества перед завершением задачи. |

## Собственные IDE-плагины

### Computer Use

Инструменты для управления рабочим столом и приложениями через визуальный интерфейс.

| Tool | Назначение | Когда использовать |
|---|---|---|
| `ComputerScreenshot` | Сделать скриншот экрана, окна или области. | Для визуальной диагностики, проверки UI, ошибок и состояния приложения. |
| `ComputerClick` | Клик по координате или найденному UI-элементу. | Когда действие доступно только через GUI. |
| `ComputerDoubleClick` | Двойной клик по элементу. | Для открытия файлов, строк таблиц, элементов списков. |
| `ComputerRightClick` | Контекстное меню по элементу. | Для действий, спрятанных в меню правой кнопки. |
| `ComputerType` | Ввод текста в активное поле. | Для форм, редакторов, поисковых строк и терминалов. |
| `ComputerHotkey` | Нажатие сочетаний клавиш. | Для быстрых команд: сохранить, найти, открыть палитру, переключить вкладку. |
| `ComputerKey` | Нажатие отдельной клавиши. | Для `Enter`, `Esc`, `Tab`, стрелок и пошаговой навигации. |
| `ComputerScroll` | Прокрутка страницы или панели. | Когда нужный элемент вне видимой области. |
| `ComputerDrag` | Перетаскивание мышью. | Для resize, drag-and-drop, canvas, карт, дизайнеров и файлов. |
| `ComputerWait` | Ожидание изменения UI. | После клика, загрузки, анимации, запуска приложения или сетевого действия. |

Минимальная защита: перед destructive-действиями в GUI нужен скриншот текущего состояния и явная проверка цели.

| Tool | Назначение | Когда использовать |

Минимальная защита: не вводить секреты в сторонние сайты без явного разрешения; не выполнять destructive-действия в продакшене без подтверждения.

Минимальная защита: любые write-действия в GitHub должны явно указывать repo, target branch/PR/issue и итоговое изменение.

## Быстрый выбор tool

Если нужно вызвать несколько tools, runtime запускает их batch-режимом: `parallel` для независимых действий, `sequence` для зависимых цепочек, `dag` для смешанных сценариев.

| Задача | Первый tool |
|---|---|
| Понять задачу и собрать нужный контекст | `FastContext` |
| Понять структуру проекта | `RepoMap` |
| Найти, где реализована логика | `SemanticSearch` |
| Найти определение, использования и связи символа | `SymbolContext` |
| Найти точную строку или regex | `Grep` |
| Найти файл по имени/маске | `Glob` |
| Найти связанные тесты, стили, типы, routes, API | `RelatedFiles` |
| Понять файл перед правкой | `Read` |
| Создать новый файл | `Write` |
| Изменить маленький фрагмент | `StrReplace` |
| Понять текущие git-изменения | `GitContext` |
| Запустить тесты или сборку | `Shell` |
| Найти правильные тесты для изменения | `TestContext` |
| Проверить ошибки проекта | `DiagnosticsContext` |
| Разобрать упавший лог | `FailureAnalyzer` |
| Проверить веб-UI | `BrowserOpen` + `BrowserScreenshot` |
| Управлять desktop-приложением | `ComputerScreenshot` + нужное UI-действие |
| Проверить CI в PR | `GitHubListChecks` + `GitHubGetCheckLogs` |
| Найти документацию по версиям проекта | `DocsContext` |
| Использовать внешнюю интеграцию | `CallMcpTool` или `FetchMcpResource` |
| Проверить итоговый diff | `ReviewDiff` |
